## 値オブジェクト

- 不変
- 同一性を持たない
- 等価性で比較される
- 完全に交換可能
- 生成時に常に妥当な状態
- 副作用を持たない

## エンティティ

- 同一性を持つ
- 時間を通じて同一である
- 状態の変化に意味がある
- 可変である
- ライフサイクルをもつ
- 交換不可能

## 値オブジェクトとエンティティの違い

| 本質 | エンティティ | 値オブジェクト |
| --- | --- | --- |
| 本質 | 可変 | 不変 |
| 識別 | 同一性(ID) | 等価性(値) |

なるべくイミュータブル(不変)に実装したい。
そうできるものを値オブジェクトとし、ライフサイクルを管理したい場合はエンティティとする。

## イミュータブルにする理由

- 保守性の向上
    - 挙動を理解しやすい
        - オブジェクトが作られた後のふるまいが固定化されるため
    - スレッドセーフ
        - 副作用(メソッドが引数を受け取り値を返す以外の状態変更すること)によってインスタンス変更がされた場合、結果が処理順に影響される
        - この例はインスタンス変数だが、グローバル変数などでイメージする場合と同じ
        - 他にも引数の変更(Byref)とか、ログ出力とかのI/Oも副作用
    - テストがし易い
        - 入力に対して出力が固定されるため、出力値テストによって検証可能

```csharp
public sealed class Money
{
    public Currency Currency { get; }
    public decimal Amount { get; }

    public Money(Currency currency, decimal amount)
    {
        Currency = currency ?? throw new ArgumentNullException(nameof(currency));

        if (amount < 0)
            throw new ArgumentException("金額は0以上である必要があります");

        Amount = amount;
    }

    public Money Add(Money other)
    {
        if (other is null)
            throw new ArgumentNullException(nameof(other));

        if (!Currency.Equals(other.Currency))
            throw new InvalidOperationException("異なる通貨同士は加算できません");

        return new Money(Currency, Amount + other.Amount);
    }
}
```

```csharp
public sealed class Money
{
    public Currency Currency { get; private set; }
    public decimal Amount { get; private set; }

    public void Add(Money other)
    {
        if (other is null)
            throw new ArgumentNullException(nameof(other));

        if (!Currency.Equals(other.Currency))
            throw new InvalidOperationException("異なる通貨同士は加算できません");

        // ❌ 自身の状態を書き換えている（不変性の破壊）
        Amount += other.Amount;
    }
}
```

## クラスにする理由

- データ(状態)とふるまいをまとめられる(凝集性を高める)
- コンストラクタによって不正な状態にさせなくできる
    - コンストラクタが複雑になる場合は、Privateメソッド＋ファクトリーメソッドとする。

### デメテルの法則

> オブジェクト*O*上のメソッド*M*が呼び出してもよいメソッドは以下のオブジェクトに属するメソッドのみに限定される
> 
> 1. *O*それ自身
> 2. *M*の引数に渡されたオブジェクト
> 3. *M*の内部でインスタンス化されたオブジェクト
> 4. *O*を直接的に構成するオブジェクト（*O*のインスタンス変数）

そもそもインスタンス変数を使用しないメソッドは、データ(状態)とふるまいをまとめるという考え方には違反している

## ドメインサービス

- 状態を持たない
- 値オブジェクトやエンティティに分類できないふるまいを定義する
- ただし多用すると、値オブジェクトやエンティティでやりたかった高凝集な実装ができなくなる(ドメインモデル貧血症)ため、まずはドメインモデルでの実装を検討する
