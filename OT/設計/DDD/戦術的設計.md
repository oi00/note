# 📦 値オブジェクト

# 1. 特徴

* **不変（Immutable）**

  * 生成後に状態は変更されない
  * 変更は新しいインスタンス生成で表現する

* **同一性を持たない**

  * IDではなく値そのもので識別される

* **等価性で比較される**

  * 値が同じであれば同一とみなす（Equals）

* **完全に交換可能**

  * 同じ値であればどこで使っても意味が変わらない

* **生成時に常に妥当な状態**

  * 不正な値は生成できない

* **副作用を持たない**

  * 外部状態を変更しない

---

# 2. 設計原則

* **コンストラクタ（またはFactory）で妥当性を保証する**

  * 不正値は生成時に排除する

* **不変性を維持する**

  * フィールドは変更不可
  * 更新は新しいインスタンスで返す

* **等価性を正しく定義する**

  * 値ベースでEquals / GetHashCodeを実装する

* **ドメイン上意味のある振る舞いのみ持つ**

  * 汎用的な処理ではなく、概念に紐づく操作のみ

* **副作用を持たないメソッド設計**

  * メソッドは新しい値を返す

* **内部で整合性を完結させる**

  * 外部にルールを漏らさない

---

# 3. 採用基準

次のいずれかに該当する場合、値オブジェクトを検討する：

* **ドメインルールを持つ値**

  * 例：金額、メールアドレス、名前

* **型で意味を強制したい**

  * プリミティブだと誤用されるもの

* **単体で扱う価値がある**

  * 他と切り離しても意味がある概念

* **等価性で扱える**

  * 「同じなら同じ」で問題ない

* **不正値を排除したい**

  * バリデーションが必須な値

---

# 4. 作る理由（設計上の価値）

* **不正な状態を構造的に排除する**

  * 「チェックする」ではなく「生成できなくする」

* **ドメインルールを局所化する**

  * ロジックの散在を防ぐ
  * 変更影響を最小化する

* **型によって誤用を防ぐ（型安全性）**

  * コンパイル時にバグを検出できる

* **ユビキタス言語をコードで表現する**

  * ただし本質は「意味と制約の強制」

* **モデルをシンプルに保つ**

  * 同一性やライフサイクル管理が不要

* **副作用を排除し、安全に扱える**

  * 参照共有が安全
  * 予測可能な動作になる
  * 並列処理でも破綻しない

---

# 🎯 最終まとめ（設計判断の軸）

> **その値は「ただのデータ」か？「ルールを持つ概念」か？**

* ただのデータ → プリミティブでよい
* ルールを持つ概念 → 値オブジェクトにする

---

# 📦 エンティティ


# 1. 特徴

* **同一性（Identity）を持つ**

  * IDによって識別される
  * 値が同じでも別物として扱う

* **時間を通じて同一である**

  * 状態が変化しても同じものとして扱う

* **状態の変化に意味がある**

  * 変更そのものがドメイン上の重要な概念

* **可変である（ただし制御される）**

  * 状態は変更されるが、無制限ではない

* **ライフサイクルを持つ**

  * 生成 → 変更 → 削除という流れがある

* **交換不可能**

  * 同じ値でも別のインスタンスは別物

---

# 2. 設計原則

* **Identityは不変にする**

  * 一度決まったIDは変更しない

* **状態変更はメソッド経由に限定する**

  * フィールドの直接変更は禁止

* **状態遷移の整合性を保証する**

  * 不正な遷移を許可しない

* **振る舞いで状態を変更する**

  * セッターではなく、意味のある操作で変更する
    （例：`Ship()`, `Cancel()`）

* **不正な状態を作らせない**

  * 常に整合性が保たれた状態を維持する

* **外部から内部状態を壊させない**

  * カプセル化を徹底する

---

# 3. 採用基準

次のいずれかに該当する場合、エンティティを検討する：

* **同一のものとして識別し続ける必要がある**

  * 例：ユーザー、注文、契約

* **状態の変化を追跡したい**

  * 変更履歴や現在状態が重要

* **ライフサイクルを持つ**

  * 生成・更新・削除の流れがある

* **値が同じでも区別する必要がある**

  * 同姓同名の人物など

* **状態遷移にルールがある**

  * 特定の順序でしか変化できない

---

# 4. 作る理由（設計上の価値）

* **同一の対象を一貫して扱うため**

  * 状態が変わっても同じものとして扱える

* **状態の変化をドメインとして表現するため**

  * 「変わること」に意味を持たせる

* **状態遷移の整合性を保つため**

  * 不正な操作を防ぐ

* **ライフサイクルを管理するため**

  * 生成から削除までを一貫して扱う

* **振る舞いと状態を結びつけるため**

  * ロジックの散在を防ぐ

---

# 🎯 最終まとめ（設計判断の軸）

> **その対象は「変化し続ける同一のもの」として扱いたいか？**

* YES → エンティティ
* NO → 値オブジェクト

---

# 🔥 値オブジェクトとの対比（最重要理解）

| 観点      | エンティティ    | 値オブジェクト |
| ------- | --------- | ------- |
| 本質      | 変化する存在    | 不変の値    |
| 識別      | 同一性（ID）   | 等価性（値）  |
| 可変性     | あり（制御される） | 不変      |
| ライフサイクル | あり        | なし      |
| 交換可能性   | 不可        | 可能      |

---

本質は、イミュータブル(不変)な実装を増やしていくことにある。
なるべくイミュータブルに出来るものはイミュータブルにする。
